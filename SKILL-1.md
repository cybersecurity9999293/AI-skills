### 1. SYSTEM DESIGN — DISTRIBUTED ARCHITECTURE

**CAP Theorem Mastery**
- CP systems (consistency + partition tolerance): PostgreSQL, ZooKeeper, etcd
- AP systems (availability + partition tolerance): Cassandra, DynamoDB, CouchDB
- Choice depends on business requirements, not personal preference

**Scalability Patterns**
```
Vertical Scaling      → CPU/RAM upgrades (ceiling exists)
Horizontal Scaling    → More instances (preferred, stateless required)
Database Sharding     → Partition data by key (user_id, tenant_id)
Read Replicas         → Separate read/write traffic
CQRS                  → Command/Query Responsibility Segregation
Event Sourcing        → Append-only log as source of truth
Saga Pattern          → Distributed transaction management
```

**Load Balancing Strategies**
- Round Robin — equal distribution, stateless services
- Least Connections — long-lived connections, WebSockets
- Consistent Hashing — cache affinity, session stickiness
- Weighted — heterogeneous instance sizes

**Caching Architecture (Multi-Layer)**
```
L1: In-process cache (application memory, microseconds)
L2: Distributed cache (Redis/Memcached, milliseconds)
L3: CDN cache (edge nodes, < 5ms globally)
L4: Database query cache (last resort)
```
Cache invalidation strategies: TTL, event-driven invalidation, write-through, write-behind, cache-aside.

### 2. DATABASE MASTERY

**Schema Design Principles**
- Normalize to 3NF first, then denormalize intentionally for read performance
- Every table needs: `id` (UUID v7), `created_at`, `updated_at`, soft-delete `deleted_at`
- Index strategy: index columns used in WHERE, JOIN, ORDER BY clauses
- Composite indexes: column order matters — highest cardinality first
- Partial indexes: `WHERE deleted_at IS NULL` saves massive space at scale

**Query Optimization Checklist**
```sql
-- ✅ Use EXPLAIN ANALYZE before shipping any complex query
EXPLAIN ANALYZE SELECT ...;

-- ✅ Avoid N+1 with proper JOINs or batch loading
-- ❌ SELECT * FROM users WHERE id = $id (per loop iteration)
-- ✅ SELECT * FROM users WHERE id = ANY($ids) (single query)

-- ✅ Pagination: Use cursor-based (keyset) not OFFSET for large datasets
-- ❌ SELECT * FROM posts LIMIT 20 OFFSET 10000 (full scan)
-- ✅ SELECT * FROM posts WHERE id > $cursor ORDER BY id LIMIT 20
```

**Connection Pooling Rules**
- Always use PgBouncer or connection pooler in production
- `max_connections = (cpu_cores * 2) + effective_spindle_count`
- Pool size per service: `(concurrent_requests / avg_query_duration_ms) * 1000`

**Migration Strategy**
- All schema changes: backward-compatible first, then cleanup
- Additive changes only (new columns, new tables) — never rename/drop in same deploy
- Zero-downtime: expand-contract pattern

### 4. SECURITY — ZERO COMPROMISE

**Authentication & Authorization**
```
Authentication Flow:
  1. User presents credentials
  2. Server validates, issues short-lived JWT (15min) + refresh token (7 days)
  3. JWT stored in memory (not localStorage), refresh in httpOnly cookie
  4. All requests authenticated via Authorization: Bearer <token>
  5. Refresh token rotation on every use (detect reuse = revoke all)

Authorization Patterns:
  RBAC  → Role-Based Access Control (user, admin, superadmin)
  ABAC  → Attribute-Based (user.tier === 'enterprise' && resource.org === user.org)
  PBAC  → Policy-Based (Cedar, OPA) for complex multi-tenant systems
```

**OWASP Top 10 — Engineering Mitigations**
```
A01 Broken Access Control     → ABAC/RBAC on every endpoint, deny by default
A02 Cryptographic Failures    → AES-256-GCM encryption at rest, TLS 1.3 in transit
A03 Injection                 → Parameterized queries ONLY, never string concatenation
A04 Insecure Design           → Threat modeling before implementation
A05 Security Misconfiguration → Infrastructure as Code, secrets in Vault/AWS Secrets Manager
A06 Vulnerable Components     → Dependabot, automated CVE scanning in CI
A07 Auth Failures             → MFA, rate limiting on auth, lockout after N attempts
A08 Software/Data Integrity   → Signed artifacts, SBOM, supply chain verification
A09 Logging Failures          → Structured logging, immutable audit trails
A10 SSRF                      → Allowlist-only outbound requests, metadata blocking
```

**Secrets Management**
```
❌ NEVER: .env files committed to git
❌ NEVER: Secrets in Docker images or container env vars visible to all
✅ ALWAYS: HashiCorp Vault / AWS Secrets Manager / GCP Secret Manager
✅ ALWAYS: Secret rotation automated
✅ ALWAYS: Audit logs on every secret access
```

**Latency Budget Framework**
```
Global SLA target: p99 < 200ms
  ├── Network: < 20ms
  ├── Load balancer: < 2ms
  ├── Application logic: < 50ms
  ├── Database queries: < 100ms
```

**Profiling-Driven Optimization**
```
Rule: NEVER optimize without a profiler.
Rule: NEVER optimize code that isn't a proven bottleneck.
Rule: Measure → Identify → Fix → Measure again.

Tools:
  Python  → cProfile, py-spy, memory_profiler
  Node.js → clinic.js, 0x, --prof flag
  Go      → pprof, trace
  Rust    → flamegraph, criterion
```

**Async Architecture Patterns**
```python
# ✅ Concurrent I/O — run independent operations in parallel
results = await asyncio.gather(
```

- Streaming for large payloads — never load 1GB files into memory
- Pagination on all list queries — never return unbounded result sets
- Weak references for cache entries to allow GC when memory pressure exists

### 7. CLEAN CODE & ARCHITECTURE PATTERNS

**SOLID Principles Applied**
```
S — Single Responsibility: One class, one reason to change
    ❌ UserService handles auth + billing + email
    ✅ AuthService, BillingService, NotificationService

O — Open/Closed: Open for extension, closed for modification
    ✅ Strategy pattern, plugin architecture

L — Liskov Substitution: Subtypes replaceable without breaking behavior
    ✅ Interface-first design

I — Interface Segregation: Many specific interfaces > one general one
    ✅ IReadableRepository, IWritableRepository vs IRepository

D — Dependency Inversion: Depend on abstractions, not concretions
    ✅ Constructor injection, IoC containers
```

**Domain-Driven Design (DDD)**
```
Strategic Patterns:
  Bounded Context  → Clear domain boundaries, no shared mutable state
  Context Map      → Explicit integration patterns between domains
  Ubiquitous Lang  → Code uses domain expert vocabulary

Tactical Patterns:
  Entity           → Has identity, mutable over time (User, Order)
  Value Object     → Defined by value, immutable (Money, Address)
  Aggregate        → Consistency boundary, single root entity
  Repository       → Collection abstraction over persistence
  Domain Event     → Something significant happened (OrderPlaced)
  Domain Service   → Business logic that doesn't belong to an entity
```

**Clean Architecture Layers**
```
┌─────────────────────────────────┐
│     Presentation Layer          │ Controllers, API handlers, CLI
├─────────────────────────────────┤
│     Application Layer           │ Use cases, command/query handlers
├─────────────────────────────────┤
│     Domain Layer                │ Entities, value objects, domain services
├─────────────────────────────────┤
│     Infrastructure Layer        │ DB, cache, external APIs, messaging
└─────────────────────────────────┘
Dependency rule: Inner layers NEVER import from outer layers
```

**Design Patterns — When to Use**
```
Creational:
  Factory         → Decouple object creation from usage
  Builder         → Complex object construction (QueryBuilder, HttpClient)
  Singleton       → Shared resource, use with caution (prefer DI)

Structural:
  Adapter         → Wrap incompatible interfaces
  Decorator       → Add behavior without modifying (middleware, logging)
  Facade          → Simplified interface to complex subsystem

Behavioral:
  Strategy        → Interchangeable algorithms (payment providers, auth methods)
  Observer        → Event-driven decoupling (domain events, webhooks)
  Command         → Encapsulate operations (undo/redo, queuing, CQRS)
  Circuit Breaker → Fail fast, prevent cascade failure
```

### 8. TESTING — ELITE COVERAGE PHILOSOPHY

**Testing Pyramid**
```
         /\
        /E2E\        5% — Critical user journeys only
       /──────\
      /Integr. \     15% — Service boundaries, DB, external APIs
     /──────────\
    /    Unit    \   80% — Pure functions, business logic, edge cases
   /______________\
```

**Unit Test Standards**
```python
# ✅ Arrange-Act-Assert, one concept per test
def test_calculate_order_total_applies_discount_for_premium_users():
    # Arrange
    user = UserBuilder().with_tier("premium").build()
    order = OrderBuilder().with_items([item(price=100)]).build()

    # Act
    total = calculate_order_total(order, user)

    # Assert
    assert total == Decimal("90.00")  # 10% premium discount applied
```

**Testing Anti-Patterns to Eradicate**
```
❌ Testing implementation details (testing private methods)
❌ Brittle assertions on exact strings / timestamps
❌ Tests that depend on execution order
❌ Mocking everything (tests that pass but code is broken in prod)
❌ No test for error paths / edge cases
❌ Test code with lower quality than production code
```

**Property-Based Testing** (for complex business logic)
```python
# Hypothesis (Python) — generate 100s of inputs automatically
@given(price=st.decimals(min_value=0, max_value=10000))
def test_discount_never_exceeds_price(price):
    discount = calculate_discount(price, tier="premium")
    assert discount <= price
```

**Contract Testing** (for microservices)
- Consumer-driven contracts with Pact
- API schema validation with OpenAPI/JSONSchema
- Database migration testing in CI before deploy

## ⬡ CODE REVIEW CHECKLIST

When reviewing any code, evaluate against this rubric:

**Correctness** — Does it work for all inputs, including edge cases?
**Security** — Does it introduce any OWASP vulnerabilities?
**Performance** — Are there obvious bottlenecks? N+1 queries? Missing indexes?
**Readability** — Can a junior engineer understand this in 2 minutes?
**Testability** — Can this be unit tested without infrastructure?
**Scalability** — Will this design survive 100x current load?
**Observability** — Will we know when and why this fails in production?
**Error Handling** — Are all failure modes handled explicitly?

```

❌ Hardcoded secrets, credentials, or magic strings
❌ Deprecated language features or outdated patterns
❌ N+1 query patterns in database access code
❌ Missing input validation on any public interface
❌ Synchronous blocking I/O in async contexts

```
